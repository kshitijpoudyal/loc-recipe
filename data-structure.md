# Data Structure: LOC Recipe

## 1. Overview

The application uses **Firebase Firestore** as its primary database, with **Firebase Cloud Storage** for recipe images and **Firebase Auth** for user management. The data model has two core collections: `recipe` and `dailySchedule`, linked by recipe IDs.

```
Firebase Project: loc-recipe
│
├── Auth            → User accounts (email/password)
├── Firestore
│   ├── recipe/     → All recipes
│   └── dailySchedule/ → Weekly meal plan (one doc per weekday)
└── Storage
    └── images/     → Recipe images
```

## 2. Core Data Models

### 2.1 Recipe

**Firestore Collection:** `recipe`
**Document ID:** Auto-generated by Firestore (`recipeId`)

```typescript
interface Recipe {
  recipeId?: string;          // Auto-generated Firestore document ID
  name: string;               // Recipe name (required)
  prepTime?: number;          // Prep time in minutes
  cookTime?: number;          // Cook time in minutes
  servings?: number;          // Number of servings
  mealType?: string[];        // ["breakfast", "lunch", "dinner"]
  ingredients?: Ingredient[]; // List of ingredients
  steps?: string[];           // Ordered cooking instructions
  ageGroup?: string[];        // ["adult", "kids"]
  nutrition?: Nutrition;      // Per-serving nutritional info
  imageUrl?: string;          // Firebase Storage download URL
  createdAt?: Date;           // Timestamp of creation
  createdBy?: string;         // Firebase Auth user UID
}
```

**Nested: Ingredient**
```typescript
interface Ingredient {
  name: string;       // e.g., "flour"
  quantity: number;   // e.g., 2
  unit: string;       // e.g., "cups"
}
```

**Nested: Nutrition** (per serving)
```typescript
interface Nutrition {
  calories?: number;      // kcal
  protein?: number;       // grams
  carbohydrates?: number; // grams
  fats?: number;          // grams
  sugar?: number;         // grams
}
```

**Example Firestore Document:**
```json
{
  "recipeId": "abc123",
  "name": "Chicken Curry",
  "prepTime": 15,
  "cookTime": 30,
  "servings": 4,

}
```

### 2.2 DailySchedule

**Firestore Collection:** `dailySchedule`
**Document ID:** Weekday value (e.g., `"monday"`, `"tuesday"`)

```typescript
interface DailySchedule {
  scheduleId?: string;   // Firestore document ID (mirrors weekday value)
  weekday: string;       // "monday" | "tuesday" | ... | "sunday"
  breakfast: string[];   // Array of Recipe document IDs
  lunch: string[];       // Array of Recipe document IDs
  dinner: string[];      // Array of Recipe document IDs
  createdBy?: string;    // Firebase Auth user UID
}
```

**Example Firestore Document (`dailySchedule/monday`):**
```json
{
  "scheduleId": "monday",
  "weekday": "monday",
  "breakfast": ["abc123", "def456"],
  "lunch": ["ghi789"],
  "dinner": ["jkl012", "mno345"],
  "createdBy": "uid_xyz789"
}
```

## 4. Derived / Computed Structures

These are not stored in Firestore — they are assembled at runtime by joining collections.

```typescript
type MappedSchedule = Record<string, Record<MealType, Recipe[]>>;
```

## 5. Firebase Utility Functions

### 5.1 Recipe (`src/utils/firebaseUtils/Recipe.ts`)

| Function | Description |
|---|---|
| `fetchAllRecipes()` | Fetches all documents from the `recipe` collection |
| `findRecipeById(id)` | Returns a single recipe by its document ID |
| `addRecipeToFirebase(recipe)` | Creates a new recipe document |
| `uploadImage(file)` | Uploads an image to Cloud Storage and returns the download URL |

### 5.2 DailySchedule (`src/utils/firebaseUtils/DailySchedule.ts`)

| Function | Description |
|---|---|
| `fetchAllDailySchedules()` | Fetches all 7 weekday schedule documents |
| `addScheduleToFirestore(schedule)` | Creates or overwrites a schedule document |
| `updateSchedule(weekday, mealType, recipeIds)` | Updates a specific meal slot for a given day |
| `mapAllRecipesToSchedule(recipes)` | Joins recipe data into the schedule, returns `MappedSchedule` |

### 5.3 User (`src/utils/firebaseUtils/User.ts`)

| Function | Description |
|---|---|
| `registerUser(email, password)` | Creates a new Firebase Auth account |
| `authenticateUser(email, password)` | Signs in with email/password |
| `getSignedInUser()` | Returns the currently authenticated Firebase user |
| `logoutUser()` | Signs out the current user |

## 6. Data Flow Diagrams

### Add Recipe Flow
```
User fills form
     │
     ▼
uploadImage(file) ──► Firebase Storage ──► returns imageUrl
     │
     ▼
addRecipeToFirebase({ ...recipe, imageUrl })
     │
     ▼
Firestore: recipe/{auto-id}
```

### Daily Schedule Flow
```
DailySchedule page loads
     │
     ├── fetchAllRecipes() ──► Firestore: recipe/*
     │
     └── fetchAllDailySchedules() ──► Firestore: dailySchedule/*
              │
              ▼
     mapAllRecipesToSchedule(recipes)
              │
              ▼
     MappedSchedule rendered as weekly grid
```

### Assign Recipe to Day Flow
```
User picks a recipe + weekday + meal type
     │
     ▼
updateSchedule(weekday, mealType, [...existingIds, newRecipeId])
     │
     ▼
Firestore: dailySchedule/{weekday}.{mealType} updated
```

## 7. Firestore Collection Summary

| Collection | Doc ID | Key Fields | Notes |
|---|---|---|---|
| `recipe` | Auto (string) | `name`, `mealType`, `createdBy` | Images stored separately in Cloud Storage |
| `dailySchedule` | Weekday string | `breakfast[]`, `lunch[]`, `dinner[]` | Stores recipe IDs, not embedded docs |

## 8. Relationships

```
User (Firebase Auth)
  │  createdBy ──────────────────────┐
  │                                  │
  ▼                                  ▼
recipe/{id}                dailySchedule/{weekday}
  - name                     - breakfast: [recipeId, ...]
  - ingredients              - lunch:     [recipeId, ...]
  - steps                    - dinner:    [recipeId, ...]
  - nutrition
  - imageUrl ──► Firebase Storage
```

- `DailySchedule` references `Recipe` by ID (normalized, not embedded).
- All user-generated data carries a `createdBy` field matching the Firebase Auth UID.
- There is no sub-collection nesting — both collections are flat at the root level.
